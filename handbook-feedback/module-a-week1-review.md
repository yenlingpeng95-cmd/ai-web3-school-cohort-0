## 📌 理解摘要

### 總體感受

模塊 A 前三章設計了一個很清楚的「認知金字塔」：

1. **LLM（Day 1）** → 理解模型的能力邊界與局限性
2. **Prompt（Day 2）** → 學習如何把任務轉成模型可執行的指令
3. **Context（Day 3）** → 理解模型看到什麼、為什麼看到這些、看不到什麼

三章互為前提，缺一不可。尤其對於產品背景的學習者來說，這套框架非常實用——它把「怎麼跟 AI 合作」從個人使用層面，提升到了系統設計層面。

---

### 📖 章節一：大語言模型（LLM）

**最深刻的 takeaways：**

1. **LLM 是概率模型，不是事實來源。** 它生成的是「概率上合理的輸出」，不是「天然可信的事實」。在任何嚴肅系統裡，模型是推理層，不能取代資料庫、API 或鏈上數據作為事實源。

2. **Embedding ≠ 事實判斷。** 向量相似度只能說明內容接近，不能替代來源校驗。這個區別對 RAG 設計非常重要——很多人以為 embedding 搜到最像的就對了，但「像」不代表「對」。

3. **越靠近執行層，模型輸出越需要結構化。** LLM 越靠近真實動作（簽名、轉帳、寫入），系統越要把自然語言變成可驗證對象（schema、參數、引用、審計日誌）。

**問題 / 不清楚的地方：**
- Handbook 中把模型定位為「推理層」，但實際產品中模型有時也承擔「分類層」或「生成層」——這三種角色在系統架構中的拆分標準是什麼？什麼時候該讓模型推理，什麼時候該用傳統規則？

**與其他章節的關聯：**
- LLM 的能力邊界直接決定了 Prompt 和 Context 的設計目標：因為模型會幻覺，所以 Prompt 需要有邊界；因為模型只看得到上下文，所以 Context 需要信息治理。

---

### 📖 章節二：提示詞（Prompt）

**最深刻的 takeaways：**

1. **Instruction 的四段結構（任務目標 → 可用輸入 → 禁止行為 → 輸出格式）** 非常實用。它不是「寫更長的 prompt」，而是把模糊任務拆成可穩定執行的規範。

2. **Few-shot 是雙面刃。** 示例能讓模型模仿風格，但協議升級或業務規則改變後，舊示例會反過來誤導模型——它需要跟 eval 一起維護，不能一次寫完就放著。

3. **Structured Output 的價值不在好看，在於可檢查。** 讓模型返回 JSON schema（如 action / risk_level / uncertainties），後續程式碼才能檢查、拒絕、記錄和回歸測試。這是從個人使用到產品級的關鍵差異。

**問題 / 不清楚的地方：**
- Prompt Injection 提到了多層防禦（內容標記 + 參數校驗 + allowlist + human approval），但實際產品中這幾層的責任何誰定義、誰維護？比如「這個動作需不需要 human approval」的規則是寫在 prompt 裡、程式碼裡、還是權限系統裡？

**與其他章節的關聯：**
- Prompt 是「軟約束」，Context 提供「可信資料」，兩者搭配才構成模型安全運作的前置條件。Prompt 定義任務，Context 提供素材。

---

### 📖 章節三：上下文（Context）

**最深刻的 takeaways：**

1. **Context 是信息治理，不是長文本拼接。** 為每類信息標註來源、時效、權限和可信度，否則模型會把「用戶說的」「網頁寫的」「鏈上查到的」「系統規定的」混在同一層處理。

2. **五層 Context 分層設計（指令層 → 任務層 → 事實層 → 知識層 → 記憶層）** 非常清晰。層級越清楚，後續越容易做權限控制、prompt injection 防護和審計。

3. **Memory 不能替代即時授權。** 用戶過去允許某個動作，不代表現在仍然允許。所有涉及身份、權限、資產或外部副作用的記憶，都必須重新綁定當前會話和當前授權。

**問題 / 不清楚的地方：**
- Hermes Agent 的 Context Engineering 具體是怎麼實作的？它的 system prompt、tools list、memory、user profile 是如何分層組織到 context 裡的？
- 未來如果要做產品級的 Agent，這五層的設計有什麼常見陷阱？特別是「事實層」與「知識層」的邊界——鏈上實時狀態該放哪一層？文件檢索結果又該放哪一層？

**與其他章節的關聯：**
- Context 是 LLM 和現實世界的橋樑。沒有好的 Context，Prompt 設計再好也沒用——模型看到的不是真實世界。反過來，Context 再好，如果 LLM 不理解自己的能力邊界，也會濫用不該用的訊息。

---

## 🔗 三章之間的貫穿主線

從第一週三章的學習中，我觀察到一條貫穿主線：

> **「模型輸出是候選結果，不是事實本身；模型能力是推理入口，不是最終驗證。」**

這條主線在每一章以不同形式出現：
- LLM 章 → 概率模型 vs 事實來源
- Prompt 章 → 軟約束 vs 安全邊界
- Context 章 → 信息治理 vs 長文本拼接

對於產品背景的我來說，這個框架的實用價值在於：**它把 AI 產品的問題從「模型還不夠強」轉移到了「系統設計夠不夠嚴謹」**——這是產品人可以著力的方向。
