## 📝 學習記錄

> 昨天學了錢包如何管理私鑰、簽署交易，今天看錢包**在跟誰互動** — 智能合約。如果說錢包是「私鑰的容器」，那智能合約可以理解為**部署在鏈上的公開程式**，Web3 產品的核心邏輯都在這裡。

---

### 🔑 核心認知：智能合約到底是什麼？

**智能合約 ≠ 自動執行的法律合約**

智能合約是**部署在區塊鏈上的程式**，把規則、資產和狀態放到公開可驗證的執行環境裡。

> 💡 **一句話記住**：智能合約把規則變成**公共基礎設施**，也把 bug 變成**公共風險**。

---

### ⚡ 最小例子：Counter 計數器

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.20;

contract Counter {
    uint256 public count;

    event CountChanged(uint256 newCount);

    function increment() external {
        count += 1;
        emit CountChanged(count);
    }
}
```

這個最小的例子已經包含合約最核心的幾件事：
| 元素 | 說明 |
|------|------|
| `count` | **鏈上狀態** — 寫入區塊鏈，永久儲存 |
| `increment()` | **寫操作** — 改變狀態，需簽名 + 付 gas |
| `event` | **事件日誌** — 外部系統可以索引的記錄 |
| `count()` (public) | **讀操作** — 讀取狀態，不需簽名，不需 gas |

> 真實協議（Uniswap、AAVE、USDC）只是把這個模型放大：餘額、抵押品、借款、訂單、投票、權限管理，都是更複雜的鏈上狀態。

---

### 🧩 1. Solidity — EVM 生態最常見的合約語言

**難度：初級** — 先用它理解鏈上程式的寫法和限制

Solidity 看起來像普通程式語言，但它運作在完全不同的環境：

| 鏈上特有概念 | 重要性 |
|--------------|--------|
| **storage** | 鏈上儲存很貴（gas 高），變數位置很重要 |
| **msg.sender** | 誰在呼叫這個函數？所有權驗證的核心 |
| **modifier** | 權限控制（如 `onlyOwner`），保護關鍵操作 |
| **event** | 讓外部系統（前端、索引器）知道發生了什麼 |
| **external call** | 呼叫其他合約有風險（重入攻擊等）|
| **revert** | 交易失敗時回滾所有狀態變更 |
| **gas** | 每次執行的運算成本，無限迴圈會被 block |

---

### 🖥️ 2. EVM（Ethereum Virtual Machine）

**難度：中級** — 理解合約如何被執行

EVM = 運行智能合約的**虛擬機器**

- Solidity 程式碼 → 編譯成 **EVM 位元組碼** → 部署到鏈上 → 由節點執行
- 解釋了為什麼要有 gas、為什麼 storage 貴、為什麼外部呼叫有風險
- 同一套合約可以部署到多個 EVM 相容鏈（ETH、Polygon、Arbitrum、Base...）

---

### 🔌 3. ABI（Application Binary Interface）

**難度：初級** — 應用和合約溝通的接口說明

- ABI 描述合約有哪些**函數、參數、回傳值、事件**
- 前端 SDK、腳本、區塊瀏覽器都靠 ABI 來理解如何呼叫合約
- **對 AI × Web3 特別重要**：Agent 透過 ABI 理解合約能力

> ⚠️ ABI 只告訴你「能呼叫什麼」，不保證「呼叫是否安全」
> - to 是不是惡意地址？
> - amount 的單位對不對（decimals）？
> - 這個函數是否被暫停或有權限限制？

**ABI = 機器可讀接口，不是安全說明書。**

---

### 📡 4. Event（事件）

**難度：初級** — 合約跟外部世界「說話」的方式

- 合約執行時可以 `emit event`
- event **不影響合約狀態**，但會記錄在鏈上交易收據中
- 非常適合讓外部系統追蹤發生過什麼：轉帳、訂單建立、參數變更

**產品中的應用**：NFT marketplace 可能的 events
```
OrderCreated
OrderCancelled
OrderFilled
```
前端訂單列表、用戶歷史、資料看板，往往不是直接掃鏈上 storage，而是**透過索引事件**來構建查詢資料庫。

---

### 🔄 5. Upgrade（合約升級）

**難度：高級** — 安全、治理與產品迭代之間的權衡

| | 不可升級 | 可升級（Proxy） |
|---|---|---|
| **優點** | 規則固定、去信任 | 可修 bug、迭代功能 |
| **缺點** | bug 難修 | 需要治理、多簽、timelock |
| **信任假設** | 不需要信任任何人 | 要信任升級管理者 |

**判斷升級風險的四個問題：**
1. 升級權限在誰手裡？單一 EOA / 多簽 / DAO / timelock？
2. 用戶能提前看到升級提案和新實作地址嗎？
3. 升級能不能改變資產轉移、提款、暫停或權限邏輯？
4. 如果管理者金鑰洩漏，最壞結果是什麼？

> 💡 如果一個協議說「合約已審計」，但沒說清楚升級權限，這個安全說明仍不完整。

---

### 🔗 一個完整呼叫流程

以「用戶點擊投票按鈕」為例：

```
1. 前端讀取錢包地址 + 目前網路
2. 前端根據 ABI 編碼 vote(proposalId) 呼叫資料
3. 錢包顯示交易資訊，用戶確認簽名
4. 交易被廣播到 RPC 節點
5. 驗證者把交易打包進區塊
6. EVM 執行合約邏輯
   → 成功：更新狀態 + 發出 event
   → 失敗：revert（狀態不變，但 gas 仍會扣）
7. 前端監聽交易收據，更新頁面狀態
8. 索引器讀取 event，更新歷史記錄或看板
```

> 智能合約開發不能只看 Solidity 檔案，要同時理解**錢包、RPC、區塊瀏覽器、前端 SDK、event 和索引層**。

---

### ❌ 常見錯誤

| 錯誤 | 說明 |
|------|------|
| 混淆 view 和寫交易 | 讀取狀態不需簽名，改變狀態需要交易 + gas |
| 忽略 decimals | Token amount 不是人類看到的「1.5」，而是放大後的整數 |
| 只測成功路徑 | 權限失敗、餘額不足、重複呼叫、過期時間、暫停狀態都要測 |
| 把 owner 當永遠可信 | Owner 權限太大時，用戶實際上在信任管理者 |
| 忽略事件設計 | 沒有 event，後續前端歷史、索引和審計會很困難 |

### 🤖 在 AI × Web3 中的位置

```
AI Agent (建議 + 編排)
    ↓
錢包 (授權確認)
    ↓
智能合約 (可驗證執行)
    ↓
監控系統 (記錄結果)
```

- AI Agent 可以幫用戶理解 ABI、生成呼叫參數、解釋交易結果、寫測試
- **但合約負責最終執行邊界** — 即使 AI 輸出不穩定，鏈上規則仍有明確邊界

---

### 🧠 核心總結

```diff
+ 智能合約 = 部署在鏈上的公開程式，不是法律合約
+ Solidity → EVM 位元組碼 → 鏈上執行
+ ABI = 機器可讀接口（告訴你「能呼叫什麼」，不保證安全）
+ Event = 外部系統追蹤鏈上變化的主要方式
+ Upgrade 是安全與靈活性的權衡，不能不說清楚
+ 讀狀態不需簽名，寫狀態需要簽名 + 付 gas
```

| 觀念 | 一句話記住 |
|------|-----------|
| 智能合約本質 | **部署在鏈上的公開程式，規則變成公共基礎設施** |
| Solidity | **鏈上特有概念比語法更重要：storage、msg.sender、modifier、event** |
| ABI 的角色 | **機器可讀接口，不是安全說明書** |
| Event | **合約跟外部世界「說話」的方式** |
| Upgrade | **可升級 ≠ 方便改 bug，而是安全與治理的權衡** |
| 完整呼叫 | **錢包簽名 → RPC → 區塊 → EVM 執行 → event 索引** |
