## 📝 學習記錄

### 學到的三件事
1. **LLM 是概率模型，不是事實來源：** LLM 生成的是「概率上合理的輸出」，不是「天然可信的事實」。它的位置是推理層，不是真相源——關鍵事實必須來自資料庫、API、鏈上數據或其他可信系統。這決定了它在任何嚴肅系統裡的角色邊界。

2. **Context 是信息治理，不是長文本拼接：** 為每類信息標註來源、時效、權限和可信度，否則模型會把「用戶說的」「網頁寫的」「鏈上查到的」「系統規定的」混在同一層處理。Context Engineering 的目標不是塞滿窗口，而是讓模型在正確的信息層級裡工作。

3. **Memory 不能替代即時授權：** 模型記住「用戶常用某地址」不等於它現在還可以用。用戶過去允許某個動作，不代表現在仍然允許。所有涉及身份、權限、資產或外部副作用的記憶，都必須重新綁定當前會話和當前授權。

### 問題 / 疑問
- Hermes 的 Context Engineering 具體是怎麼實作的？它的 system prompt、tools list、memory、user profile 是如何分層組織到 context 裡的？未來如果要做產品級的 Agent，這些層的設計有什麼常見陷阱？

### 關鍵概念
- **LLM 基礎：** Token → Embedding → Transformer → Hallucination → Multimodal
- **Context Window：** 128K tokens（DeepSeek V4 Flash），越大不等於越好，需配合檢索與摘要
- **Context Engineering：** 數據源選擇、排序、裁剪、標註來源、隔離不可信內容
- **Memory vs Knowledge Base：** Memory 跨請求保留（偏好/歷史），KB 外部可檢索（文檔/規範）
- **五層 Context 分層：** 指令層 → 任務層 → 事實層 → 知識層 → 記憶層
