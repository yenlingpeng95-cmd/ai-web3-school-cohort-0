## 📝 學習記錄

### 學到的三件事
1. **Instruction 的四段結構：** 實用的 prompt 不是寫長篇咒語，而是拆成「任務目標 → 可用輸入 → 禁止行為 → 輸出格式」，讓模型可以穩定執行
2. **Few-shot 是雙面刃：** 放示例能讓模型模仿風格和判斷邏輯，但協議升級或業務規則改變後，舊示例會反過來誤導模型——它需要跟 eval 一起維護
3. **Structured Output 不是為了好看：** 固定 schema（如 action / risk_level / uncertainties）讓後續程式碼能檢查、拒絕、記錄和回歸測試，這才是產品級 prompt 的核心

### 問題 / 疑問
- 今天的實作環節提到「用 Hermes 試寫不同 prompt 風格」——我其實每天都在用 Hermes，但不太確定有沒有刻意從 prompt 工程的角度去設計它。這算不算已經在實作了？還是我應該刻意寫一個實驗性的 prompt 出來？

### 關鍵概念
- Instruction（任務規則的四段結構）
- Few-shot（示例引導，需與 eval 一起維護）
- Structured Output（JSON schema / 函數參數）
- Prompt Injection（外部內容不可信、工具參數校驗、human approval）
- Prompt 不該獨自承擔安全 → 需搭配 Context + Model + Code + Guard + Human 的完整鏈路
